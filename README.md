# Техническое задание: Микросервис авторизации и личного кабинета пользователей

Данный документ представляет собой техническое задание (ТЗ) на разработку бэкенд-модуля для регистрации, аутентификации и управления сессиями пользователей. Документация подготовлена для передачи команде разработки.

---

## 1. Функциональные требования (User Stories)

Для описания логики системы используются Пользовательские истории (User Stories) и критерии их приемки (Acceptance Criteria).

### US-1: Регистрация нового пользователя
**Как** Гость системы,  
**Я хочу** зарегистрироваться по номеру телефона и паролю,  
**Чтобы** получить доступ к закрытым функциям платформы.

*   **Критерии приемки (Acceptance Criteria):**
    *   **AC 1.1 (Валидация телефона):** Номер телефона должен вводиться строго в международном формате: `+7XXXXXXXXXX` (всего 11 цифр). При неверном формате система выдает ошибку: *"Неверный формат номера телефона"*.
    *   **AC 1.2 (Валидация пароля):** Пароль должен быть длиной не менее 8 символов, содержать минимум одну заглавную букву и одну цифру. При нарушении — ошибка: *"Пароль слишком простой"*.
    *   **AC 1.3 (Уникальность):** Если пользователь с таким номером телефона уже существует в базе данных, система блокирует регистрацию и отдает ошибку: *"Пользователь с таким номером уже зарегистрирован"*.
    *   **AC 1.4 (Результат):** При успешной регистрации в базе данных создается новая запись, пароль сохраняется в зашифрованном виде (хэш), а пользователю отправляется статус успешного завершения.

### US-2: Авторизация (Вход в систему)
**Как** Зарегистрированный пользователь,  
**Я хочу** ввести свой номер телефона и пароль,  
**Чтобы** войти в свой личный кабинет.

*   **Критерии приемки (Acceptance Criteria):**
    *   **AC 2.1 (Успешный вход):** При совпадении пары телефон/пароль система генерирует сессионный токен доступа (JWT) и перенаправляет пользователя в личный кабинет.
    *   **AC 2.2 (Неверные данные):** При вводе неверного телефона или пароля система возвращает ошибку: *"Неверный логин или пароль"*. (Из соображений безопасности система не должна уточнять, что именно введено неверно).
    *   **AC 2.3 (Безопасность):** При вводе неверного пароля более 5 раз подряд для одного номера телефона, учетная запись временно блокируется на 15 минут. Выводится сообщение: *"Превышено число попыток входа. Доступ заблокирован на 15 минут"*.

## 2. Модель данных (Структура БД и SQL-запросы)

Для хранения данных пользователей и обеспечения безопасности сессий спроектированы две таблицы в реляционной СУБД (PostgreSQL). 
Схема связи между таблицами: **users (1) -------- (M) sessions** (Один-ко-Многим). Один пользователь может иметь несколько активных сессий с разных устройств.

### Структура таблицы `users` (Данные пользователей)


| Имя поля | Тип данных | Ограничения | Описание |
| :--- | :--- | :--- | :--- |
| `id` | SERIAL | Primary Key | Уникальный ID пользователя (автоинкремент) |
| `phone` | VARCHAR(15) | Unique, Not Null | Номер телефона (используется как логин) |
| `password_hash`| VARCHAR(255)| Not Null | Захэшированный пароль для безопасного хранения |
| `created_at` | TIMESTAMP | Not Null, Default NOW() | Дата и время регистрации пользователя |

### Структура таблицы `sessions` (Активные сессии устройств)


| Имя поля | Тип данных | Ограничения | Описание |
| :--- | :--- | :--- | :--- |
| `id` | SERIAL | Primary Key | Уникальный ID сессии |
| `user_id` | INT | Foreign Key | ID пользователя (ссылка на users.id) |
| `token` | VARCHAR(500)| Unique, Not Null | JWT-токен для доступа к закрытому API |
| `expires_at` | TIMESTAMP | Not Null | Срок годности токена (дата истечения сессии) |

### SQL-скрипты для разработчиков

```sql
-- 1. Запрос для проверки данных при авторизации (поиск по телефону)
SELECT id, phone, password_hash 
FROM users 
WHERE phone = '+79991112233';

-- 2. Запрос на получение всех активных сессий конкретного пользователя
SELECT u.phone, s.token, s.expires_at
FROM sessions s
JOIN users u ON s.user_id = u.id
WHERE s.user_id = 1 AND s.expires_at > NOW();
```
